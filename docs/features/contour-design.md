# Contour — Design Concepts

> **ARCHIVED** — содержимое перенесено в `docs/features/contours.md` (2026-05-29).
> Этот файл оставлен для истории решений. Актуальный дизайн — в `contours.md`.

**Date**: 2026-04-25

Концептуальные решения, принятые при проектировании фичи Контур. Здесь зафиксированы
инварианты и обоснования, которые не отражены напрямую в коде или плане реализации.

---

## Контур и транспорт — неразрывная связь

Контур без транспорта теряет смысл и превращается в пустую коробку для хранения.
Транспорт — это то, что делает контур рабочим: определяет канал передачи, PSK,
идентификатор для routing'а входящих пакетов.

**Инварианты:**
- Транспорт генерируется автоматически при создании контура
- Пользователь не может удалить транспорт контура
- Пользователь может опосредованно менять только некоторые настройки (например, имя контура)
- В коде: `transport: ContourTransport` — non-nullable поле; контур без транспорта не создаётся

---

## Слот — зона ответственности ноды, не контура

Контур не знает, на каком слоте он находится. Это решается на уровне ноды при синхронизации.

**Принцип**: контур хранит только параметры канала (имя, PSK, hash). Слот назначается
динамически при connect: приложение ищет первый свободный слот и записывает туда канал.

**Следствие**: нет поля `slotIndex` в `Contour`. `ChannelSlotResolver` строит маппинг
`slot → hash` в runtime из данных ноды. При получении пакета: `slot → hash → Contour`.

**Обработка нехватки слотов**: MVP — только `Log.w`, без UI. Обработка отложена (TODO).

## ContourTransport — структура для расширения

```kotlin
data class ContourTransport(
    val meshtastic: MeshtasticChannel,
    // future: val satellite: SatelliteChannel? = null
)
```

Текущий этап — один транспорт (`MeshtasticChannel`). Структура выбрана явной (не `List`),
чтобы при добавлении новых протоколов просто добавлять поля, не меняя контракт.

---

## Emergency контур — hardcoded вне DB

Emergency контур (`"Emergency"`, slot 0, open PSK) полностью определён в коде.
В DB не хранится. `isActive` персистируется в DataStore.

**Обоснование**: изменение Emergency контура не предусмотрено проектом. Hardcoded объект
исключает случайное изменение или удаление через общий CRUD-путь.

**Следствие для `observeContours()`**: репозиторий всегда prepend'ит Emergency в список,
подставляя актуальный `isActive` из DataStore.

---

## Default контур — обычный deletable контур

`DefaultActiveContour` ("Basic") — seeded при первом запуске, но ничем не
отличается от пользовательских контуров. Может быть удалён. После удаления не
восстанавливается автоматически.

**Обоснование**: нет смысла защищать его как системный — пользователь должен иметь
возможность построить собственную структуру контуров.

---

## expiration — только UI, не влияет на routing

Поле `expiration: Instant?` означает срок актуальности контура. После истечения:
- UI показывает контур как устаревший в списке
- Routing и доставка сообщений продолжают работать без изменений
- `isActive` не сбрасывается автоматически

Полезно для контуров, созданных под разовые мероприятия.

---

## exclusivityTime — концепция (реализация отложена)

Поле `exclusivityTime: Instant?` — timestamp, до которого активность контура должна быть
эксклюзивной.

**Семантика**: если у контура задан `exclusivityTime` и он ещё не истёк, приложение
должно деактивировать все остальные контуры на время действия эксклюзивности.

**Пример использования**: спортивные соревнования, где участники разных групп должны
быть разделены по каналам — включение контура группы A автоматически отключает группы B, C.

**Текущий статус**: поле присутствует в модели и хранится в DB, но логика эксклюзивности
не реализована в MVP. Отображение в UI — отложено.
