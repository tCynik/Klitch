# Node Provisioning Auto-Config

## What it does
Приложение само настраивает геопозиционные параметры ноды при подключении ("включил, нажал кнопку, забыл") — пользователь не должен разбираться в специфике протокола Meshtastic. `LocationConfigCard` показывает текущее состояние ноды только как информацию, без ручного редактирования.

## Key classes
- `NodeProvisioningUseCase.provisionPositionConfig()` (domain) — при каждом подключении к ноде проверяет, совпадают ли position-настройки с firmware-дефолтами; если да — пишет Klitch-преcет (`gpsMode=DISABLED`, `broadcastIntervalSecs=LocationConfigDefaults.APP_DRIVEN_BROADCAST_SECS` = `Int.MAX_VALUE`, `smartBroadcastEnabled=false`, `positionFlags=897`, `provideLocationToMesh=true`, `primaryChannelPositionPrecision=32`). Если уже сконфигурировано — пропускает запись (без лишнего ребута ноды).
- `NodeProvisioningUseCase` — стале `fixedPositionEnabled` снимается безусловно (`removeFixedPosition`), независимо от firmware-default guard.
- `NodeProvisioningUseCase` — `needsLegacyMigration`-guard: ноды, провижененные до фикса `position-broadcast-interval-unification.md` (застрявшие на старом `1800`), детектируются через собственный fingerprint (`gpsMode=DISABLED && provideLocationToMesh=true`) + старое значение, и переписываются на правильный preset один раз.
- `LocationConfigCard` (presentation) — read-only отображение всех секций (provide-to-mesh, gpsMode, fixed position, broadcast interval, smart broadcast, position flags, channel precision). Единственный интерактивный элемент — `useWakeLock` (локальная настройка телефона, не нода).
- `GpsService.onCreate()` — переехавший туда `requestIgnoreBatteryOptimizationIfNeeded()` (раньше был привязан к ручному тогглу `provideLocationToMesh` в `NetworkSettingsScreen`).

## Non-obvious decisions
- **`broadcastIntervalSecs=Int.MAX_VALUE` — прошивка не транслирует автономно вообще**, позицию шлёт только приложение (`sendPosition`, см. `phone-gps-to-radio.md`) — прошивка штампует `current_time` на каждой автономной ре-трансляции, что ложно "освежает" метку у соседей даже после отключения телефона (см. `gps-position-staleness.md`).
- **`BROADCAST_INTERVAL_HIGH` warning — mode-aware, не universal-threshold**: `LocationConfigUi.sharingStatus` сравнивает `broadcastIntervalSecs` с ожидаемым значением по `gpsMode` — `LocationConfigDefaults.APP_DRIVEN_BROADCAST_SECS` для `DISABLED` (PHONE_GPS), `PositionTrackingPolicy.STATIONARY_INTERVAL_SECS` (mesh-модуль) для `ENABLED` (NODE_GPS), не проверяется для `NOT_PRESENT`. `LocationConfigDefaults.APP_DRIVEN_BROADCAST_SECS` ([LocationConfigModel.kt](app/src/main/java/ru/tcynik/klitch/domain/mesh/model/LocationConfigModel.kt)) — общая константа, читается `NodeProvisioningUseCase`, `MeshConfigRepositoryImpl.prepareNodeForAppDrivenBroadcast()`/`disableNodePositionBroadcast()`, `SyncContoursOnConnectUseCase` и `LocationConfigUi` — единый источник, см. `docs/plans/position-broadcast-interval-unification.md`.
- **Конфликт `NodeProvisioningUseCase` ↔ `SyncContoursOnConnectUseCase`**: оба пишут `position_broadcast_secs`, но разными путями (`NodeProvisioningUseCase` — одним protobuf-сообщением вместе с `gpsMode`/`flags`; `SyncContoursOnConnectUseCase` — через `prepareNodeForAppDrivenBroadcast()`). Делегация в общий call-site не выбрана — два последовательных admin-write на одну ноду означали бы два ребута подряд. Вместо этого оба пишут одно и то же значение константы — конфликта по значению больше нет, реализации остаются раздельными.
- **`GPS_MODE_CONFLICT` warning удалён** — раньше `gpsMode==ENABLED` считался конфликтом (нода со своим GPS могла "перезаписать" позицию телефона). После того как `node-gps-position-source.md` ввёл `PositionSourceMode.NODE_GPS` как полноценный параллельный режим (телефонный мост явно не запускается), это больше не конфликт — просто другой валидный режим со своим cadence-preset.
- **`useWakeLock` остался editable**: это локальная настройка телефона (PARTIAL_WAKE_LOCK), не параметр протокола Meshtastic — не противоречит принципу "юзер не должен разбираться в протоколе".
- **idempotent guard расширен новыми полями** (`provideLocationToMesh`, `primaryChannelPositionPrecision`) в тот же `isFirmwareDefault` чек, что и старые (broadcast/smart/flags) — гарантирует один write-cycle на свежей ноде, без повторных ребутов на каждый коннект.
- **`fixedPositionEnabled` removal — отдельная безусловная проверка**, не часть `isFirmwareDefault`: блокер может появиться независимо от провижининга (например, нода была сконфигурена через официальный Meshtastic app).
- **`provideLocationToMesh` — тоже отдельная безусловная проверка** (баг-фикс: раньше входила в `isFirmwareDefault` bundle — если нода уже провижена по остальным полям, но тоггл выключен вручную вне приложения, например в официальном Meshtastic app, весь гейт фейлился и тоггл не включался обратно никогда). Теперь при каждом коннекте: если `!provideLocationToMesh` → `setProvideLocation(true)` сразу, независимо от состояния прочих полей. `needsLegacyMigration` fingerprint упрощён до `gpsMode=DISABLED && broadcastIntervalSecs==1800` (без завязки на `provideLocationToMesh`, т.к. оно больше не индикатор состояния провижининга).

## Known limitations / planned extensions
- Auto-config срабатывает только через `NodeProvisioningUseCase.provision()`, вызываемый из `ConnectionViewModel` при `onNodeConnected()`. Если нода была переконфигурена вручную (например, в официальном приложении) и параметры не совпадают с firmware-дефолтами — auto-config не сработает повторно (намеренно, чтобы не перетирать ручные изменения и не вызывать лишние ребуты).
