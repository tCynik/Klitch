# MeshTactics — Каналы и идентичность пользователя

> Архитектурные решения, выработанные в дискуссии 2026-04-15 и 2026-04-20.
> Основа для планирования рефакторинга domain-слоя.

---

## Ключевой принцип

**Приложение — источник истины. Нода — тупое радио.**

Нода не определяет ни идентичность пользователя, ни структуру каналов.
Приложение конфигурирует ноду на основе своих настроек, а не наоборот.

---

## Логический канал (LogicalChannel)

### Определение

Логический канал — это **именованное пространство видимости** на уровне логики приложения.
Всё, что происходит внутри канала (сообщения, позиции участников, метки, телеметрия),
видно только участникам этого канала.

Логический канал не зависит от транспорта. Транспорт — деталь реализации.

### Структура

```
LogicalChannel
├── id: LogicalChannelId            — хэш учётных данных (детерминированный)
├── name: String                    — отображаемое название (позывной группы, операции)
├── metadata: ChannelMetadata       — расширяемые метаданные уровня приложения
└── transports: List<TransportBinding>
         └── MeshtasticBinding
                  ├── channelIndex: Int   — индекс слота на ноде (0–7)
                  └── psk: ByteArray      — учётные данные Meshtastic-канала
         └── WifiBinding (будущее)
                  └── ...credentials...
```

Учётные данные (PSK и пр.) хранятся **внутри транспортных байндингов**, а не на уровне `LogicalChannel`.
`LogicalChannel` — контейнер: объединяет транспорты, метаданные и идентичность.

### Метаданные канала (ChannelMetadata)

Расширяемый набор атрибутов уровня приложения. Управляется только приложением,
не зависит от транспорта. Состав определяется в ходе разработки.

Примеры атрибутов (не финальные):
- цвет отображения на карте
- иконка / символ
- описание / назначение
- роль в операции (командный, разведывательный, логистический…)

Реализация: типизированный класс с известными полями + возможность расширения.
Не `Map<String, String>` — чтобы не терять типобезопасность на известных полях.

### Идентификация канала через хэш учётных данных

`LogicalChannelId = hash(транспортные credentials)` — детерминированный идентификатор.

**Свойство:** одни и те же учётные данные → один и тот же ID на всех устройствах участников.
Не требует синхронизации UUID out-of-band. Участник с теми же credentials автоматически
попадает в тот же канал в SQLDelight-хранилище.

>

### Расширяемость транспортов

Сейчас реализован только Meshtastic. В будущем добавятся `WifiBinding`, `MqttBinding`.
Приоритет между транспортами будет настраиваться отдельно (облик не сформирован).

### Множественные каналы

Пользователь может быть участником нескольких логических каналов одновременно.
Автоматическая ретрансляция информации между каналами не предусмотрена.

## Профиль пользователя (AppUser)

### Определение

Идентичность пользователя живёт на уровне приложения, а не на уровне ноды.

```
AppUser
├── displayName: String     — отображаемое имя (позывной)
│                             синхронизируется в LongName ноды при подключении
└── shortName: String       — короткое имя (1–4 символа)
                              синхронизируется в ShortName ноды при подключении
                              настраивается отдельно от displayName
```

### Принцип "личность следует за человеком, а не за железом"

Один пользователь может иметь несколько нод и использовать их попеременно.
При подключении к любой ноде приложение пушит `displayName` → `LongName` ноды.
Остальные участники канала видят имя владельца аккаунта, а не имя ноды.

**Сценарий с общим пулом нод:** если у группы есть несколько общих нод,
участнику не нужно помнить, где чья нода — достаточно подключиться к любой,
и на карте появится его имя.

### Коллизия имён при нескольких одновременно активных нодах

Сценарий (один пользователь запустил несколько нод одновременно) — отложен,
не является блокирующим для MVP.

## Node Provisioning Flow

Логика при подключении ноды к приложению:

```
1. Нода подключена по BLE
2. Читаем текущую конфигурацию каналов ноды
3. Сравниваем с настройками активных LogicalChannel в приложении
4. Если расхождение:
   └── запрашиваем разрешение у пользователя
5. Записываем каналы на ноду (setChannel для каждого LogicalChannel)
6. Отключаем слоты ноды, не предусмотренные настройками приложения
7. Синхронизируем LongName ← AppUser.displayName
8. Синхронизируем ShortName ← AppUser.shortName
```

## Последствия для domain-слоя

### Что нужно изменить

| Текущее состояние | После рефакторинга |
|---|---|
| `ContactType.CHANNEL` идентифицируется строкой `"^all"` (Meshtastic-специфика) | Идентифицируется `LogicalChannelId` (хэш credentials) |
| `SendChatMessageParams.channel: Int` (индекс Meshtastic-канала) | `SendChatMessageParams.channelId: LogicalChannelId` |
| `ChatContact.id` = contactKey из Meshtastic | `ChatContact.id` = `LogicalChannelId` для каналов |

### Новые domain-модели

- `LogicalChannel` — основная абстракция канала
- `TransportBinding` (sealed interface) + `MeshtasticBinding`
- `AppUser` — профиль пользователя (источник истины об имени)
- `NodeProvisioningUseCase` — оркестрация настройки ноды при подключении

### Правило изоляции (без изменений)

Транспортная специфика (Meshtastic `channelIndex`, `psk` формат) остаётся в `app/data/`.
Domain-слой работает только с `LogicalChannelId` и `AppUser`.

## Приложение как источник истины для истории

### Принцип хранения входящих сущностей

Все локально хранимые полученные сущности (сообщения, геометки) помечаются `logicalChannelId`.

Поток получения:
```
[нода] пакет прибыл
  → data-слой декодирует
  → channelIndex → LogicalChannelId (через MeshtasticBinding)
  → INSERT в SQLDelight с logicalChannelId (идемпотентно по id пакета)
  → domain читает ТОЛЬКО из SQLDelight
```

Нода является транспортом событий, а не хранилищем состояния.

### Фильтрация по каналу

Сущности из канала, не настроенного в приложении (`logicalChannelId` не найден), **не вставляются** в SQLDelight и не показываются в интерфейсе.

Правило: «не знаю этот канал — не храню».

### Дедупликация

Meshtastic ретранслирует пакеты — один и тот же пакет может прийти несколько раз.
Защита: `INSERT OR IGNORE` по stable id пакета (UUID геометки, message id).

## Управление TTL геометок

TTL управляется приложением на основе значения `expires_at`, полученного из `Waypoint.expire` при декодировании.

### Правила

| Момент | Правило |
|---|---|
| Получение пакета | Если `expires_at <= now` — не вставлять, отбросить |
| Чтение из SQLDelight | Фильтровать `WHERE expires_at IS NULL OR expires_at > now` |
| Периодическая очистка | `deleteExpired` — запускать при старте и по таймеру (интервал TBD) |

**Устаревшие метки не хранятся.** Поле `expires_at = NULL` означает отсутствие срока — метка хранится бессрочно до явного удаления.

`Waypoint.expire = 0` трактуется как "нет срока" → `expires_at = NULL` в SQLDelight.

Значение `Waypoint.expire` используется как начальное `expires_at`. Приложение может в будущем переопределять TTL независимо от ноды (например, продлевать срок вручную). Это намеренная точка расширения.

### История геометок vs текущее состояние

Сейчас геометки — это «живые объекты» с TTL. История обмена (кто, когда, что отправил) — отдельная задача, не входящая в MVP. Когда потребуется, добавляется отдельная таблица `geo_mark_event` без TTL-фильтрации.
