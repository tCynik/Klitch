# MeshTactics — Каналы и идентичность пользователя

> Архитектурные решения, выработанные в дискуссии 2026-04-15.
> Основа для планирования рефакторинга domain-слоя.

---

## Ключевой принцип

**Приложение — источник истины. Нода — тупое радио.**

Нода не определяет ни идентичность пользователя, ни структуру каналов.
Приложение конфигурирует ноду на основе своих настроек, а не наоборот.

---

## Логический канал (LogicalChannel)

### Определение

Логический канал — это **именованное пространство видимости** на уровне логики приложения.
Всё, что происходит внутри канала (сообщения, позиции участников, метки, телеметрия),
видно только участникам этого канала.

Логический канал не зависит от транспорта. Транспорт — деталь реализации.

### Структура

```
LogicalChannel
├── id: UUID                        — внутренний идентификатор
├── name: String                    — название (позывной группы, операции)
├── psk: ByteArray                  — ключ шифрования (хранится в приложении)
└── transports: List<TransportBinding>
         └── MeshtasticBinding
                  ├── channelIndex: Int   — индекс слота на ноде (0–7)
                  └── psk: ByteArray      — ключ для Meshtastic (может совпадать с channel.psk)
```

### Расширяемость транспортов

Сейчас реализован только Meshtastic. В будущем добавятся `WifiBinding`, `MqttBinding`.
Приоритет между транспортами будет настраиваться отдельно (облик не сформирован).

### Множественные каналы

Пользователь может быть участником нескольких логических каналов одновременно.
Автоматическая ретрансляция информации между каналами не предусмотрена.

## Профиль пользователя (AppUser)

### Определение

Идентичность пользователя живёт на уровне приложения, а не на уровне ноды.

```
AppUser
├── displayName: String     — отображаемое имя (позывной)
│                             синхронизируется в LongName ноды при подключении
└── shortName: String       — короткое имя (1–4 символа)
                              синхронизируется в ShortName ноды при подключении
                              настраивается отдельно от displayName
```

### Принцип "личность следует за человеком, а не за железом"

Один пользователь может иметь несколько нод и использовать их попеременно.
При подключении к любой ноде приложение пушит `displayName` → `LongName` ноды.
Остальные участники канала видят имя владельца аккаунта, а не имя ноды.

**Сценарий с общим пулом нод:** если у группы есть несколько общих нод,
участнику не нужно помнить, где чья нода — достаточно подключиться к любой,
и на карте появится его имя.

### Коллизия имён при нескольких одновременно активных нодах

Сценарий (один пользователь запустил несколько нод одновременно) — отложен,
не является блокирующим для MVP.

## Node Provisioning Flow

Логика при подключении ноды к приложению:

```
1. Нода подключена по BLE
2. Читаем текущую конфигурацию каналов ноды
3. Сравниваем с настройками активных LogicalChannel в приложении
4. Если расхождение:
   └── запрашиваем разрешение у пользователя
5. Записываем каналы на ноду (setChannel для каждого LogicalChannel)
6. Отключаем слоты ноды, не предусмотренные настройками приложения
7. Синхронизируем LongName ← AppUser.displayName
8. Синхронизируем ShortName ← AppUser.shortName
```

## Последствия для domain-слоя

### Что нужно изменить

| Текущее состояние | После рефакторинга |
|---|---|
| `ContactType.CHANNEL` идентифицируется строкой `"^all"` (Meshtastic-специфика) | Идентифицируется `LogicalChannelId` (UUID) |
| `SendChatMessageParams.channel: Int` (индекс Meshtastic-канала) | `SendChatMessageParams.channelId: LogicalChannelId` |
| `ChatContact.id` = contactKey из Meshtastic | `ChatContact.id` = `LogicalChannelId` для каналов |

### Новые domain-модели

- `LogicalChannel` — основная абстракция канала
- `TransportBinding` (sealed interface) + `MeshtasticBinding`
- `AppUser` — профиль пользователя (источник истины об имени)
- `NodeProvisioningUseCase` — оркестрация настройки ноды при подключении

### Правило изоляции (без изменений)

Транспортная специфика (Meshtastic `channelIndex`, `psk` формат) остаётся в `app/data/`.
Domain-слой работает только с `LogicalChannelId` и `AppUser`.
